## 3. Real-Time Factor (RTF) 분석

> RTF = 오디오 길이 / 총 소요 시간. 1.0 미만이면 실시간 처리 불가.

| 단계 | 최장 오디오 | 해당 요청 총 소요 | RTF | 여유 |
|:---:|----------:|:--------------:|:---:|:---:|
| 1개 | 4.4s | 2,000ms | **2.20x** | 충분 |
| 2개 | 9.0s | 3,851ms | **2.34x** | 충분 |
| 3개 | 9.8s | 5,175ms | **1.89x** | 양호 |
| 4개 | 10.2s | 6,191ms | **1.65x** | 주의 |
| 5개 | 10.3s | 7,553ms | **1.36x** | 경계 |

RTF 2.34x(2개) → 1.36x(5개) 로 하강. 5개 동시에서 RTF 1.36x — 1.7B 모델의 5개 동시(1.33x)와 유사한 수준.

## 5. 종합 결론 및 권장 운영 기준

### 5.1 성능 요약

```
TTFA 추이 (0.6B):           총 소요 추이 (0.6B):
1개  450ms  (기준)           1개  2,000ms (기준)
2개  293ms  -35%  ← 역전     2개  3,455ms +73%
3개  376ms  -16%             3개  4,651ms +133%
4개  495ms  +10%  ↑상승 전환  4개  5,375ms +169%
5개  611ms  +36%             5개  6,881ms +244%
6개  ---- 서버 crash ----
```

**4개 이상부터 TTFA 상승 기울기가 전환됨** (3→4개: -16% → +10%).

### 5.2 운영 권장 기준 (RTX 3090 단일 GPU, 0.6B 모델)

| 구분 | 동시 요청 수 | 근거 |
|------|:-----------:|------|
| 최적 (안정+효율) | **2~3개** | TTFA 최저 수준(-35%~-16%), RTF 1.89x 이상, 총 소요 증가 완만 |
| 허용 최대 | **4~5개** | 성공률 100%, RTF 1.36x 이상 — TTFA +36% 이하 허용 가능 수준 |
| 금지 | **6개 이상** | 6개에서 서버 hang → crash 확인 |

### 5.3 추가 관찰

- **TTFA 역전**: 2~3개 동시 요청에서 단일 요청보다 TTFA가 낮음. 배치로 묶일 때 첫 토큰 생성이 효율화되는 0.6B 모델 특성
- **속도 우위**: 총 소요 시간은 모든 구간에서 0.6B > 1.7B (더 빠름) — 오디오 생성 속도 자체는 0.6B가 우세
- **안정성 상한**: 6개에서 서버 crash. 이후 7개는 Connection refused로 서버 재기동 필요. 1.7B 대비 더 낮은 임계점 보유 가능성
